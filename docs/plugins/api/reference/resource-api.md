# 📘 `resource.api` Reference

> **What this reference covers:** The full `resource.api` surface used by `ApiPlugin` native CRUD routes

**Audience:** Teams using resource-level API policy and native resource routes
**Time to read:** 15 min
**Difficulty:** Intermediate

---

## Overview

`resource.api` is where you attach API behavior directly to a resource.

Use it when you want the resource itself to describe:

- who can access each native operation
- which fields are exposed to each audience
- which fields are always hidden
- which fields can be written on `POST`, `PUT`, and `PATCH`
- whether the resource exposes native bulk create
- how the resource should appear in generated API docs
- which extra custom routes live next to the resource

This keeps policy and behavior close to the resource instead of spreading them across plugin config, custom handlers, and middleware.

---

## Full Shape

```javascript
await db.createResource({
  name: 'users',
  attributes: {
    id: 'string|optional',
    name: 'string|required',
    email: 'string|required|email',
    role: 'string|optional',
    tokenHash: 'string|optional',
    metadata: 'object|optional'
  },
  api: {
    description: 'User directory and profile endpoints',

    guard: {
      list: (ctx) => {
        ctx.setPartition('byTenant', { tenantId: ctx.user.tenantId });
        return true;
      },
      get: true,
      create: ['admin'],
      patch: ['admin', 'support']
    },

    protected: [
      'metadata.internal',
      { path: 'tokenHash', unlessRole: ['admin'] }
    ],

    views: {
      public: {
        auto: true,
        priority: 1,
        fields: ['id', 'name']
      },
      support: {
        auto: true,
        whenRole: ['support'],
        priority: 50,
        fields: ['id', 'name', 'email']
      },
      admin: {
        auto: true,
        whenRole: ['admin'],
        priority: 100,
        fields: ['id', 'name', 'email', 'role', 'tokenHash', 'metadata'],
        protected: [{ path: 'metadata.internal', unlessRole: ['admin'] }]
      }
    },

    writable: ['name', 'email'],
    readonly: ['role'],

    write: {
      create: [
        { whenRole: ['admin'], priority: 100, writable: ['name', 'email', 'role'] },
        { whenRole: ['support'], priority: 10, writable: ['name', 'email'] }
      ],
      patch: [
        { whenRole: ['admin'], priority: 100, writable: ['name', 'email', 'role'] },
        { whenRole: ['user'], priority: 10, writable: ['name', 'email'], readonly: ['role'] }
      ]
    },

    bulk: {
      create: {
        path: '/bulk',
        maxItems: 100,
        mode: 'partial'
      }
    },

    'GET /summary': async (c, { resource }) => {
      const total = await resource.count();
      return c.json({ total });
    }
  }
});
```

## Native CRUD Coverage

`resource.api` is applied only to native resource routes generated by `ApiPlugin`.

| Native route | `guard` | `views` | `protected` | `write` | Notes |
|--------------|---------|---------|-------------|---------|-------|
| `GET /resource` | Yes | Yes | Yes | No | `?view=name` supported |
| `GET /resource/:id` | Yes | Yes | Yes | No | `meta.view` added when a view is selected |
| `POST /resource` | Yes | Optional response view | Yes | Yes | `?view=name` can shape the response body |
| `POST /resource/bulk` | Yes | Optional response view | Yes | Yes | Enabled via `api.bulk.create`; returns a batch envelope |
| `PUT /resource/:id` | Yes | Optional response view | Yes | Yes | Full replacement semantics |
| `PATCH /resource/:id` | Yes | Optional response view | Yes | Yes | Partial merge semantics |
| `DELETE /resource/:id` | Yes | No | No | No | Authorization only |
| `HEAD`, `OPTIONS` metadata routes | Method dependent | No | No | No | Standard API metadata behavior |

Important:

- `views` shape the response body, not the stored record.
- `protected` always runs after projection.
- `write` only affects request payload validation for create/update/patch.
- custom routes declared inside `resource.api` are not auto-wrapped by `views`, `protected`, or `write`; apply those explicitly in the handler when needed.

---

## Evaluation Order

### Read routes

For `GET /resource` and `GET /resource/:id`:

1. `guard` decides whether the operation is allowed.
2. The plugin resolves the active view:
   - explicit via `?view=name`
   - automatic via matching `auto: true` or `default: true` views
3. `fields` applies inclusive projection.
4. `omit` removes selected paths.
5. global `protected` and view-level `protected` remove final sensitive fields.

### Write routes

For `POST`, `PUT`, and `PATCH`:

1. `guard` decides whether the operation is allowed.
2. The plugin resolves the active write policy for the operation.
3. `readonly`, `readOnly`, and `deny` reject forbidden paths.
4. If `writable` or `allow` exists, any path outside that allowlist is rejected.
5. The request returns `400 FIELD_WRITE_NOT_ALLOWED` when any path is rejected.
6. After persistence, the response may still be shaped by `views` and `protected`.

## Top-Level Keys

| Key | Type | Applies to | Purpose |
|-----|------|------------|---------|
| `description` | string | OpenAPI/docs | Human description for the resource |
| `guard` | guard config | Native CRUD | Per-operation authorization |
| `protected` | string[] or rule[] | Native CRUD responses | Final field filtering |
| `views` | object | Native CRUD responses | Audience-specific projections |
| `writable` | string[] | Create/update/patch | Global allowlist fallback |
| `readonly` / `readOnly` | string[] | Create/update/patch | Global denylist fallback |
| `write` | object | Create/update/patch | Per-operation write policy |
| `bulk` | object | Resource-native batch routes | Native bulk operations such as create |
| `'METHOD /path'` | function | Resource custom routes | Extra resource-level route handlers |

Reserved policy keys:

- `guard`
- `protected`
- `description`
- `readonly`
- `readOnly`
- `writable`
- `write`
- `views`
- `bulk`

Any other key shaped like `"METHOD /path"` is treated as a custom route definition.

## `description`

Use `description` to improve generated API docs for that resource.

```javascript
api: {
  description: 'Customer accounts managed by support and self-service flows'
}
```

This description is used by generated documentation instead of relying only on inferred names.

## `guard`

`guard` controls whether the actor may execute each native operation.

```javascript
api: {
  guard: {
    list: (ctx) => {
      ctx.setPartition('byTenant', { tenantId: ctx.user.tenantId });
      return true;
    },
    get: true,
    create: ['admin'],
    patch: ['admin', 'support'],
    delete: ['admin']
  }
}
```

Supported operation keys:

- `list`
- `get`
- `create`
- `update`
- `patch`
- `delete`

Use cases:

- tenant isolation
- owner checks
- partition scoping
- role-based CRUD restrictions

For guard syntax and advanced patterns, see [Guards](/plugins/api/guides/guards.md).

## `protected`

`protected` removes fields from API responses after projection.

### Simple form

```javascript
api: {
  protected: ['password', 'apiKey', 'metadata.internal']
}
```

### Conditional form

```javascript
api: {
  protected: [
    'internalNotes',
    { path: 'tokenHash', unlessRole: ['admin'] },
    { path: 'ssoToken', unlessScope: ['tokens:read'] }
  ]
}
```

### Rule keys

| Key | Type | Meaning |
|-----|------|---------|
| `path` / `field` | string | Field path, supports dot notation |
| `whenRole` | string or string[] | Hide only for these roles |
| `unlessRole` | string or string[] | Hide for everyone except these roles |
| `whenScope` | string or string[] | Hide only for these scopes |
| `unlessScope` | string or string[] | Hide for everyone except these scopes |

Notes:

- Applies to list, get, create response, put response, and patch response.
- Does not affect direct `resource.get()` or other non-HTTP access.
- Wins over projected `fields`.

## `views`

`views` define response shapes by audience.

```javascript
api: {
  views: {
    public: {
      auto: true,
      priority: 1,
      fields: ['id', 'name']
    },
    admin: {
      auto: true,
      whenRole: ['admin'],
      priority: 100,
      fields: ['id', 'name', 'email', 'role'],
      protected: [{ path: 'tokenHash', unlessRole: ['admin'] }]
    }
  }
}
```

### View keys

| Key | Type | Meaning |
|-----|------|---------|
| `fields` | string[] | Inclusive projection |
| `omit` | string[] | Remove projected fields |
| `protected` | string[] or rule[] | Final field filtering for this view |
| `guard` | guard config | Extra access check for that view |
| `whenRole` / `unlessRole` | string or string[] | Role matching |
| `whenScope` / `unlessScope` | string or string[] | Scope matching |
| `auto` | boolean | Allow automatic selection |
| `default` | boolean | Alias for automatic fallback behavior |
| `priority` | number | Higher value wins between matching automatic views |

### Selection rules

1. If `?view=name` is present, that exact view is requested.
2. Otherwise the plugin looks for matching `auto: true` / `default: true` views.
3. The highest `priority` wins.
4. If no view matches, the raw record continues to `protected`.

### Response metadata

When a view is selected, the API response includes:

```json
{
  "meta": {
    "view": "admin"
  }
}
```

### Common patterns

- public view + admin view
- explicit admin detail view via `?view=admin`
- actor-aware automatic selection
- compact list shape with richer detail shape

For design guidance, see [Resource Policies Guide](/plugins/api/guides/resource-policies.md).

## Global `writable` / `readonly`

Use these when the same write rule should apply across all write operations unless an operation-specific rule overrides it.

```javascript
api: {
  writable: ['name', 'email'],
  readonly: ['role', 'tenantId']
}
```

Behavior:

- `readonly` / `readOnly` reject matching paths
- `writable` allows only matching paths
- operation-specific `write.create`, `write.update`, or `write.patch` overrides the global allowlist when it defines its own `writable` / `allow`
- global readonly values still participate in rejection

---

## `write`

Use `write` for per-operation mutability.

### Static policy

```javascript
api: {
  write: {
    patch: {
      readonly: ['role', 'apiKey']
    }
  }
}
```

### Actor-aware policy

```javascript
api: {
  write: {
    patch: [
      {
        whenRole: ['admin'],
        priority: 100,
        writable: ['phone', 'role', 'isActive']
      },
      {
        whenRole: ['user'],
        priority: 10,
        writable: ['phone'],
        readonly: ['role', 'isActive']
      }
    ]
  }
}
```

### Operation keys

- `create`
- `update`
- `patch`

### Rule keys

| Key | Type | Meaning |
|-----|------|---------|
| `readonly` / `readOnly` | string[] | Reject writes to these paths |
| `deny` | string[] | Alias for readonly |
| `writable` / `allow` | string[] | Allow only these paths |
| `guard` | guard config | Extra access check before the rule matches |
| `whenRole` / `unlessRole` | string or string[] | Role matching |
| `whenScope` / `unlessScope` | string or string[] | Scope matching |
| `priority` | number | Higher value wins among matching rules |

### How matching works

1. The plugin looks only at the current operation.
2. A single object is used directly.
3. An array is filtered by actor match, then sorted by `priority`.
4. The highest-priority matching rule wins.
5. If `writable` exists, anything outside it is rejected.
6. `readonly` and `deny` always reject matching paths.

## `bulk`

Use `bulk` for native batch routes that still belong to the resource contract.

### `bulk.create`

```javascript
api: {
  bulk: {
    create: {
      path: '/bulk',
      maxItems: 100,
      mode: 'partial'
    }
  }
}
```

This enables `POST /resource/bulk` without changing the semantics of `POST /resource`.

### Supported keys

| Key | Type | Meaning |
|-----|------|---------|
| `path` | string | Route path under the resource. Default: `/bulk` |
| `maxItems` | number | Request item limit. Default: `100` |
| `mode` | `'partial' \| 'all-or-nothing'` | Process all items or stop after the first failure |

### Request shape

The route accepts either:

- a top-level JSON array
- an object with `items: []`

### Response behavior

- all success: `201`
- one or more item failures: `207`
- malformed bulk payload: `400`

Each successful item still uses:

- `guard.create`
- `write.create`
- normal resource validation and persistence
- response shaping through `views` and `protected`

Important:

- `all-or-nothing` is fail-fast, not rollback-capable
- bulk create is still resource-native; if the endpoint composes multiple resources, prefer a custom route

## Resource-Level Custom Routes

You can colocate custom routes with the resource by adding keys shaped like `"METHOD /path"` inside `resource.api`.

```javascript
await db.createResource({
  name: 'orders',
  attributes: {
    id: 'string|optional',
    total: 'number|required'
  },
  api: {
    description: 'Order endpoints',
    guard: { list: ['support', 'admin'] },

    'GET /summary': async (c, { resource }) => {
      const total = await resource.count();
      return c.json({ total });
    },

    'POST /reindex': async (c, { resource, database }) => {
      return c.json({ ok: true, resource: resource.name, hasDatabase: Boolean(database) });
    }
  }
});
```

### Custom route handler signature

```javascript
async (c, { resource, database }) => {
  // return Response, or any JSON-serializable value
}
```

### Behavior

- valid methods: `GET`, `POST`, `PUT`, `PATCH`, `DELETE`, `HEAD`, `OPTIONS`
- path must start with `/`
- returning a `Response` uses it directly
- returning any other value wraps it in the standard success envelope
- returning `undefined` or `null` results in `204 No Content`

Important:

- custom routes do not automatically inherit `views`, `protected`, or `write`
- if a custom route needs policy shaping, apply it explicitly in the handler
- use native CRUD plus `resource.api` policy when the route is still “about one resource”
- use plugin-level custom routes when the endpoint orchestrates multiple resources or external systems

For route precedence and plugin-level route config, see [Routing](/plugins/api/reference/routing.md).

## Errors

| Code | Status | Meaning |
|------|--------|---------|
| `INVALID_VIEW` | 400 | Requested view name does not exist |
| `VIEW_FORBIDDEN` | 403 | Actor is not allowed to use that view |
| `FIELD_WRITE_NOT_ALLOWED` | 400 | Submitted payload contains forbidden paths |
| `INVALID_BULK_PAYLOAD` | 400 | Bulk request body is malformed or empty |
| `BULK_LIMIT_EXCEEDED` | 400 | Bulk request exceeds the configured item limit |

## Recommended Structure

For most resources, this shape stays maintainable:

```javascript
api: {
  description: 'Human description for docs',
  guard: { ... },
  protected: [ ... ],
  views: { ... },
  write: { ... },
  'GET /summary': async (c, ctx) => { ... }
}
```

Recommendation:

- `guard` for row access
- `views` for audience payloads
- `protected` for secrets and invariants
- `write` for mutability
- custom routes for resource-adjacent operations not covered by CRUD
